# Информационная безопасность и госконтур · FireWatch

Документ описывает контур ИБ платформы для развёртывания в интересах ДЧС РК:
что уже реализовано в коде, и что предстоит закрыть перед боевой аттестацией.
Статусы: ✅ реализовано · 🟡 частично · ⬜ план (требует решения заказчика/инфраструктуры).

---

## 1. Персональные данные (ПДн)

**Правовая база:** Закон РК «О персональных данных и их защите» № 94-V, требования
по локализации (хранение и обработка ПДн граждан РК — на территории РК).

**Какие ПДн обрабатываются:**
- учётные записи сотрудников (`users`: логин, ФИО, роль, район);
- ФИО инспекторов (`inspectors`);
- контактные данные ответственных лиц из оперативных карточек (`operational_cards.extracted` →
  поля `contacts`, телефоны), извлекаемые ИИ из документов объектов.

| Мера | Статус | Где в системе |
|---|---|---|
| Хранение в БД на территории РК | ⬜ | Размещение инстанса PostgreSQL в дата-центре РК (сейчас демо — Railway); для пилота — гособлако/ЦОД РК |
| Минимизация: пароли не хранятся в открытом виде | ✅ | PBKDF2-HMAC-SHA256, 200k итераций, соль на запись — `api/app/auth.py` |
| Разграничение доступа к ПДн по ролям и районам | ✅ | RBAC + district scoping (раздел 2) |
| Журналирование доступа | ✅ | `audit_log` (раздел 3) |
| Шифрование канала (TLS) | 🟡 | TLS терминируется на Cloudflare/реверс-прокси; для контура РК — TLS до прикладного узла |
| Шифрование БД «at rest» | ⬜ | Включить шифрование тома/PostgreSQL TDE в целевом ЦОД |
| Согласие/уведомление субъектов ПДн | ⬜ | Организационная мера на стороне ДЧС |

## 2. Аутентификация и разграничение доступа

**Аутентификация:** JWT (HS256, TTL 7 дней), пароли — PBKDF2 (см. выше).
Секрет подписи — `JWT_SECRET` (вынести в секрет-менеджер целевого контура).

**Ролевая модель (RBAC):**

| Роль | Доступ |
|---|---|
| `inspector` | надзорный инспектор — **только свой район** |
| `supervisor` | руководитель управления — **только свой район** |
| `leadership` | руководство (замминистра) — весь город |
| `admin` | администратор — весь город, все модули |

**Разграничение по районам (district scoping)** — ✅ реализовано и покрыто тестами:
- район зашит в JWT при входе (`api/app/auth.py::create_token`);
- инспектор/руководитель управления **на сервере** ограничены своим районом в
  `/buildings` и `/overview` (`api/app/access.py`), обойти клиентским
  фильтром `?district=` нельзя (тест `test_inspector_cannot_bypass_scope_via_filter`);
- руководство/админ видят весь город.

**Предстоит (⬜):** двухфакторная аутентификация (2FA) для админ-ролей; ротация
`JWT_SECRET`; интеграция с ведомственным каталогом (LDAP/СУДИР) при необходимости.

## 3. Аудит действий

✅ Таблица `audit_log` (миграция `0002_audit_and_district`) фиксирует:
вход/неуспешный вход и каждый изменяющий запрос (POST/PUT/PATCH/DELETE) —
пользователь, роль, действие, путь, HTTP-код, IP, метка времени.

- Реализация: middleware `audit_mutations` + явный аудит логина
  (`api/app/main.py`, `api/app/routers/auth.py`, `api/app/audit.py`).
- Запись аудита **best-effort**: сбой журналирования не ломает бизнес-операцию,
  но логируется в stderr (не «тихий» отказ).
- Покрыто тестом `test_login_is_audited`.

**Предстоит (⬜):** неизменяемость журнала (WORM/перенос в защищённый
лог-коллектор), срок хранения по регламенту ДЧС, алертинг на аномалии.

## 4. Резервное копирование и восстановление

| Параметр | Цель | Статус |
|---|---|---|
| Полный бэкап БД | ежедневно, `pg_dump`/снапшот тома | ⬜ настроить в целевом ЦОД |
| PITR (WAL-архив) | непрерывно | ⬜ |
| Хранение бэкапов | ≥ 30 дней, отдельный носитель/регион | ⬜ |
| Учения по восстановлению | ежеквартально | ⬜ |
| **RPO** (допустимая потеря данных) | ≤ 24 ч (с PITR — минуты) | целевой |
| **RTO** (время восстановления) | ≤ 4 ч | целевой |

Схема воспроизводима из миграций (`alembic upgrade head`), данные — из бэкапа;
ML-модель пересобирается из кода детерминированно (фикс. seed), артефакты в бэкапе не нужны.

## 6. Аттестация и соответствие (дорожная карта)

Для боевого применения в гос. ИС требуется аттестация по требованиям ИБ РК:

1. ⬜ Размещение в аттестованном защищённом контуре (гособлако/ЦОД РК).
2. ⬜ Определение класса защищённости ИС и модели угроз (совместно с ДЧС/уполномоченным органом).
3. ⬜ Применение сертифицированных СЗИ (СКЗИ, межсетевое экранирование, антивирус).
4. ⬜ Аттестационные испытания и заключение.
5. 🟡 Управление секретами (`JWT_SECRET`, `ANTHROPIC_API_KEY`, строка БД) — вынести из
   `.env` в секрет-менеджер контура.
6. ⬜ Внешний пентест и сканирование зависимостей перед запуском.
