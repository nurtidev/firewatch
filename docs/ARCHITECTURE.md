# FireWatch — Архитектура и план реализации

> Предиктивная аналитика пожарной безопасности для ДЧС РК.
> Пилот: Астана, ~250 000 зданий. Инфраструктура: **Railway + Cloudflare + Claude (Anthropic)**.

Статус: код с нуля. Этот документ — отправная архитектура и дорожная карта.

---

## 1. Общая архитектура

Монорепозиторий, 4 деплоя на Railway + БД:

```
                         ┌─────────────────┐
            Cloudflare → │  Frontend (web) │  Next.js + MapLibre GL
                         └────────┬────────┘
                                  │ REST/tRPC
                         ┌────────▼────────┐
                         │  Backend API    │  NestJS (Node) или FastAPI
                         │  auth, routes,  │
                         │  chat, cards    │
                         └───┬────────┬────┘
                  внутр. RPC │        │ SQL
              ┌──────────────▼──┐  ┌──▼──────────────────┐
              │  ML-сервис      │  │ PostgreSQL + PostGIS │
              │  XGBoost + SHAP │  │ здания, гидранты,    │
              │  FastAPI        │  │ инциденты, риски     │
              └─────────────────┘  └──────────────────────┘
                                  ▲
                         ┌────────┴────────┐
                         │ Routing/изохроны│  OSRM или OpenRouteService
                         └─────────────────┘

  Внешние: Claude API (extraction + chat), Overpass API (импорт OSM),
           Object storage (Railway volume / Cloudflare R2) для PDF-сканов.
```

### Сервисы Railway
| Сервис | Технология | Назначение |
|--------|-----------|------------|
| `web` | Next.js 15 (App Router) + MapLibre GL + Tailwind/shadcn | Все 6 экранов UI |
| `api` | NestJS + Prisma **или** FastAPI + SQLAlchemy | Бизнес-логика, auth, чат-оркестрация, extraction |
| `ml`  | Python FastAPI + XGBoost + SHAP | Обучение и инференс риска, SHAP-объяснения |
| `routing` | OSRM (Docker, граф Астаны) | Время прибытия, изохроны, «слепые зоны» |
| `db` | PostgreSQL 16 + PostGIS (образ `postgis/postgis`) | Геоданные и аналитика |

> **Решение по языку backend.** Если команда сильнее в Python — делаем `api` и `ml` на FastAPI (один стек, проще делить код фич). Если в TS — `api` на NestJS, `ml` отдельным Python-сервисом. Рекомендация для скорости пилота: **FastAPI для api+ml, Next.js для web.**

## 2. Модель данных (PostgreSQL + PostGIS)

Ключевые таблицы:

```sql
buildings            -- импорт OSM: geom(Polygon), osm_id, addr, type,
                     --   year_built, floors, material, area
incidents            -- история пожаров 3 года: building_id, ts, severity, cause
risk_scores          -- building_id, score 0-100, computed_at, model_version
risk_explanations    -- building_id, feature, shap_value (для модуля 02)
operational_cards    -- building_id, raw_pdf_url, extracted_json, status
prescriptions        -- card_id, text, deadline_days, severity (модуль 03)
hydrants             -- geom(Point), status, last_check (модуль 05)
fire_stations        -- geom(Point), name, vehicles_count
inspectors           -- name, district, ...
inspection_routes    -- inspector_id, date, ordered building_ids (модуль 04)
inspection_visits    -- route_id, building_id, status, photos, checklist
chat_sessions / chat_messages  -- модуль 06, с привязкой источников
users                -- роли: inspector / supervisor / admin
```

Геоиндексы (GiST) на `geom` обязательны — иначе запросы «здания в радиусе», изохроны и «слепые зоны» будут медленными.

## 3. Разбивка по модулям

### 01 · CORE — Карта риска
- Импорт зданий из OSM (Overpass) для Астаны → `buildings`.
- Тайлы: бесплатные OSM-тайлы / собственный tile-server позже. Рендер — MapLibre GL.
- Слой риска: цвет по `risk_scores.score` (зелёный 0–35 / жёлтый 36–70 / красный 71–100).
- Фильтры: тип, район, уровень риска, дата последней проверки.
- Клик по зданию → боковая карточка (оперданные, история инцидентов, предписания).
- **Производительность:** при 250K зданий не грузить все полигоны разом — vector tiles / кластеризация / запрос по bbox видимой области.

### 02 · ML — Прогноз (XGBoost + SHAP)
- 30+ признаков: тип, материал, возраст, этажность, плотность застройки, инциденты в радиусе, погода, сезонность, праздники.
- Обучение оффлайн на `incidents` (3 года). Артефакт модели → volume/R2.
- Ежедневный пересчёт по всем зданиям (Railway cron) → `risk_scores`.
- SHAP-вклад каждого признака → `risk_explanations` (для панели «почему 87/100»).
- Метрика: ROC-AUC на отложенной выборке.

### 03 · AI — Оперкарточки (Claude extraction)
- Загрузка скана/PDF → storage.
- OCR при необходимости (если скан-картинка) → текст.
- Claude со **structured output** по форме МЧС РК ОК-1: адрес, тип, конструкция, системы пожаротушения, гидрант, ответственное лицо, эвакуация.
- Side-by-side: оригинал ↔ извлечённые поля, правка в один клик.
- Автогенерация предписаний из выявленных уязвимостей → `prescriptions`.
- **Гарантия «не выдумывает»:** жёсткий промпт + поля только из документа + флаг уверенности.

### 04 · OPS — План инспекций
- Утренний (07:00) cron: формирует маршрут на день на каждого инспектора.
- Приоритезация: уровень риска × срок с последней проверки × близость к маршруту.
- Решение маршрута — TSP-приближение через `routing` (OSRM matrix) + жадный/2-opt.
- Мобильный интерфейс инспектора: чек-лист, фото, отметка нарушений.
- Дашборд руководителя: статус выполнения в реальном времени.

### 05 · INFRA — Инфраструктура пожаротушения
- Слои: гидранты (статус, дата проверки), пожарные части (число машин).
- Изохроны прибытия от частей с учётом дорожной сети (OSRM `isochrone`/`table`).
- Автоподсветка «слепых зон»: где время прибытия > 10 мин (норматив).
- Аналитика: дефицит гидрантов, рекомендации по новым постам.

### 06 · CHAT — ИИ-аналитик
- Естественный язык → запрос к данным. Подход: **text-to-SQL** поверх read-only представлений PostgreSQL (безопаснее RAG для табличных данных).
- Строгое грунтование: ответ только из данных ДЧС, в каждом ответе — источники (какие записи/фильтры).
- Действия: «сформируй план», «экспорт в PDF/Excel», «отправить инспекторам».
- Защита: read-only роль БД, белый список таблиц, лимит на возвращаемые строки.

## 4. Дорожная карта (MVP → пилот)

**Фаза 0 — Фундамент (неделя 1)**
- Монорепо, CI, Railway-проект, PostgreSQL+PostGIS, скелеты `web`/`api`.
- Импорт зданий Астаны из OSM, базовая карта с зданиями.

**Фаза 1 — CORE + ML (недели 2–4)**
- Риск-оценки (сначала эвристика-заглушка, затем XGBoost), слой риска, карточка здания.
- ML-сервис, обучение на исторических данных, SHAP-панель, ежедневный cron.

**Фаза 2 — AI-карточки + CHAT (недели 4–6)**
- Extraction-пайплайн (Claude), экран оперкарточек, автопредписания.
- Чат text-to-SQL с источниками и экспортом.

**Фаза 3 — OPS + INFRA (недели 6–8)**
- OSRM, изохроны, «слепые зоны», карта инфраструктуры.
- Генератор маршрутов, мобильный экран инспектора, дашборд руководителя.

**Фаза 4 — Полировка к демо (неделя 8+)**
- Auth/роли, наполнение демо-данными Астаны, стабилизация, домен через Cloudflare.

> Презентация обещает «30 дней после LOI» до готовности пилота — это про развёртывание на данных ДЧС, при уже готовом прототипе. Прототип (фазы 0–4) — это и есть та работа, которую делаем сейчас.

## 5. Деплой на Railway — что конкретно нужно

1. **Репозиторий с кодом** (сейчас пуст). Монорепо; каждый сервис — отдельный Railway service с указанием root-директории.
2. **PostgreSQL + PostGIS** — отдельным сервисом из образа `postgis/postgis:16-3.4` (managed-Postgres Railway не гарантирует PostGIS). Включить расширения: `postgis`, `postgis_topology`.
3. **Сборка:**
   - `web`, `api` (Node) — Nixpacks автоматически.
   - `ml`, `routing` — собственный `Dockerfile` (XGBoost-зависимости, граф OSRM).
4. **Переменные окружения** (Railway Variables):
   - `ANTHROPIC_API_KEY` — Claude.
   - `DATABASE_URL` — на PostGIS-сервис (reference variable).
   - `OSRM_URL`, `OVERPASS_URL`, секреты auth (`JWT_SECRET`), ключ object storage.
5. **Volume** — для PDF-сканов и артефакта ML-модели (или Cloudflare R2).
6. **Cron** (Railway cron):
   - ежедневный пересчёт риска (`ml`);
   - утренний 07:00 — генерация маршрутов (`api`).
7. **Домены:** Railway-домен на `web` → Cloudflare proxy → `firewatch.dchs.kz`. API за тем же доменом по пути `/api` или поддомен `api.`.
8. **Среды:** `production` + `staging` через Railway environments.

> Инфраструктуру на Railway (проект, сервисы из репо, БД, переменные, домены, cron) можно поднять автоматически через Railway-интеграцию, как только будет код в репозитории.

## 6. Открытые вопросы (решить до старта)
- Язык backend: единый Python (FastAPI) vs Node (NestJS) + Python ML. → рекомендация: **FastAPI + Python ML**.
- Источник зданий: OSM на пилоте, замена на 2ГИС/госкадастр «по запросу» (слайд 05).
- Хранение данных ДЧС: юридические требования РК к локализации (упомянуто в приложениях презентации) — влияет на регион Railway / необходимость on-prem.
- OCR-движок для сканов (Tesseract vs облачный) — зависит от качества карточек.
- Реальные исторические данные для ML появятся только после LOI; до этого — синтетика/демо.
